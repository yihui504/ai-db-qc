## Summary

When executing a vector search with top_k=0, Milvus experiences a TYPE-3 crash (severe error) instead of properly validating the input. This is a critical stability issue.

## Reproduction Steps

1. Create a collection and insert data
2. Build and load index
3. Execute a search operation with top_k=0
4. Observe the error response

## Expected Result

Search should be rejected with clear validation error (e.g., 'top_k must be > 0'). System should remain stable and operational.

## Actual Result

System experiences TYPE-3 crash with unhandled exception or segmentation fault. May require service restart. Error classification: TYPE-3 (crash/failure).

## Root Cause

The search operation does not validate top_k parameter before using it. Code likely uses top_k in array indexing or buffer allocation without checking for zero values.

## Impact

- System crashes on invalid input
- Service may become unavailable
- Potential for denial-of-service attacks
- Unhandled crashes may corrupt in-memory state

## Workaround

- Always validate top_k on client side before sending to Milvus
- Use a monitoring service to auto-restart crashed Milvus instances
- Implement circuit breakers to prevent repeated failures

## Proposed Fix

- Add input validation for top_k parameter (check if not isinstance(top_k, int) or top_k <= 0)
- Add comprehensive input validation at API boundaries
- Add error handling tests for boundary values (0, 1, -1)
- Add monitoring/alerting for validation failures
- Document valid parameter ranges in API docs
