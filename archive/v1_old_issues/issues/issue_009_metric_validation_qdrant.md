## Summary

Qdrant accepts invalid or unsupported distance metric types when creating collections or indexes without proper validation.

## Reproduction Steps

1. Attempt to create a collection with an invalid distance metric
2. Observe whether the operation is rejected
3. If accepted, attempt search operations

## Expected Result

Invalid metrics should be rejected with clear error messages. Only supported metrics should be accepted.

## Actual Result

Invalid metrics may be accepted without clear validation error. May cause runtime issues.

## Root Cause

Metric type validation may be missing or insufficient. May only check string format, not semantic validity. Whitelist of supported metrics not enforced.

## Impact

- Invalid configurations cause unpredictable behavior
- No immediate feedback on incorrect configuration
- Failures occur later making them harder to diagnose

## Workaround

- Use only well-documented metrics (Cosine, Euclid, Dot, Manhattan)
- Check Qdrant documentation
- Validate on client side

## Proposed Fix

- Implement strict metric validation with whitelist
- Document all supported metrics
- Add validation tests
- Improve error messages with list of supported metrics
